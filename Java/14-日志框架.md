# Java 日志框架

**日志（Logging）** 是程序运行时的"黑匣子"。出问题时没有日志 = 闭着眼修车。专业项目都用规范的日志框架，而不是 `System.out.println`。

## 一、为什么不用 System.out.println？

```java
// ❌ 新手常见写法
System.out.println("用户 " + userId + " 登录成功");
```

**问题：**
- ❌ 没有**级别区分**（信息、警告、错误一团乱）
- ❌ 没有**时间戳**（不知道什么时候打的）
- ❌ 没有**线程信息**（多线程下完全混乱）
- ❌ 不能输出到**文件**，更不能**轮转/归档**
- ❌ 上线后想关掉？只能改代码重新部署
- ❌ **性能差**（每次都直接 IO）

```log
// ✅ 用日志框架后
2026-05-08 14:23:18.452 [http-nio-8080-exec-1] INFO  c.e.UserService - 用户 12345 登录成功
```

## 二、Java 日志生态地图

```
日志门面（接口）           日志实现（具体框架）
─────────────────         ─────────────────
SLF4J          ←→         Logback     ⭐ 现代主流
Commons Logging          Log4j 2     ⭐ 性能强
                         Log4j 1.x   （已停止维护）
                         JUL（java.util.logging）
                         System.out  （别用了）
```

**为什么有"门面"？**

业务代码只依赖 **SLF4J 接口**，底层换 Logback 还是 Log4j 2 都不影响代码——**面向接口编程**的经典应用。

## 三、SLF4J + Logback —— 现代标配

**这套组合是目前 Java 生态的事实标准**，Spring Boot 默认就用它。

### Maven 依赖

```xml
<!-- 只要引入 logback，SLF4J 会自动作为依赖被引入 -->
<dependency>
    <groupId>ch.qos.logback</groupId>
    <artifactId>logback-classic</artifactId>
    <version>1.5.6</version>
</dependency>
```

> 💡 Spring Boot 项目**不用自己引**，starter 已经包含了。

## 四、第一个日志程序

```java
import org.slf4j.Logger;
import org.slf4j.LoggerFactory;

public class UserService {
    // 每个类一个 logger，传入当前类的 Class
    private static final Logger log = LoggerFactory.getLogger(UserService.class);

    public void login(String username) {
        log.info("用户 {} 尝试登录", username);

        try {
            // 业务逻辑
            log.debug("开始查询用户信息");
            // ...
            log.info("用户 {} 登录成功", username);
        } catch (Exception e) {
            log.error("用户 {} 登录失败", username, e);
        }
    }
}
```

输出：
```
2026-05-08 14:23:18.452 [main] INFO  c.e.UserService - 用户 alice 尝试登录
2026-05-08 14:23:18.453 [main] DEBUG c.e.UserService - 开始查询用户信息
2026-05-08 14:23:18.500 [main] INFO  c.e.UserService - 用户 alice 登录成功
```

## 五、日志级别（重点！）

从低到高：

| 级别        | 用途                                   |
| --------- | ------------------------------------ |
| **TRACE** | 最详细的跟踪，几乎不用                          |
| **DEBUG** | 开发调试用，生产关闭                           |
| **INFO**  | 关键业务节点（**生产环境默认级别**）                 |
| **WARN**  | 警告，潜在问题，但不影响运行                       |
| **ERROR** | 错误，业务无法继续                            |

### 级别的过滤机制

**只有 ≥ 配置级别**的日志才会被输出。

```
配置 = INFO
则输出：INFO, WARN, ERROR
不输出：TRACE, DEBUG
```

### 怎么选合适的级别？

```java
log.trace("进入方法 a()");                     // 几乎不用
log.debug("查询参数：{}", param);              // 调试细节
log.info("订单 {} 创建成功", orderId);          // 关键业务节点
log.warn("库存不足，剩余 {} 件", stock);          // 提醒人关注但不报错
log.error("支付失败，订单 {}", orderId, ex);     // 真出错了
```

## 六、占位符 `{}` —— 必学技巧

### ❌ 错误的字符串拼接

```java
log.info("用户 " + userId + " 余额：" + balance);
```

**问题：**
- 即使日志级别关闭，**字符串拼接也会执行**（性能损失）
- `+` 拼接对象会调用 `toString()`，可能 NPE

### ✅ 正确的占位符

```java
log.info("用户 {} 余额：{}", userId, balance);
```

**优势：**
- **懒拼接**：日志级别关闭时不拼接，性能更好
- 可读性更高
- 自动处理 `null`

### 异常日志的特殊写法

```java
try {
    // ...
} catch (Exception e) {
    // ❌ 不会打印堆栈！
    log.error("操作失败：" + e.getMessage());

    // ✅ 异常对象作为最后一个参数（不需要占位符）
    log.error("操作失败", e);

    // ✅ 也可以带消息和参数
    log.error("订单 {} 处理失败", orderId, e);
}
```

> 💡 **异常对象是日志方法的"特殊参数"**——放在最后，不占用 `{}`。

## 七、Logback 配置文件

在 `src/main/resources/` 下创建 `logback.xml`：

```xml
<?xml version="1.0" encoding="UTF-8"?>
<configuration>

    <!-- 控制台输出 -->
    <appender name="CONSOLE" class="ch.qos.logback.core.ConsoleAppender">
        <encoder>
            <pattern>%d{yyyy-MM-dd HH:mm:ss.SSS} [%thread] %-5level %logger{36} - %msg%n</pattern>
        </encoder>
    </appender>

    <!-- 文件输出 -->
    <appender name="FILE" class="ch.qos.logback.core.rolling.RollingFileAppender">
        <file>logs/app.log</file>
        <rollingPolicy class="ch.qos.logback.core.rolling.TimeBasedRollingPolicy">
            <!-- 按天归档，保留 30 天 -->
            <fileNamePattern>logs/app.%d{yyyy-MM-dd}.log</fileNamePattern>
            <maxHistory>30</maxHistory>
            <!-- 单文件最大 100MB -->
            <maxFileSize>100MB</maxFileSize>
            <!-- 总大小限制 3GB -->
            <totalSizeCap>3GB</totalSizeCap>
        </rollingPolicy>
        <encoder>
            <pattern>%d{yyyy-MM-dd HH:mm:ss.SSS} [%thread] %-5level %logger{36} - %msg%n</pattern>
        </encoder>
    </appender>

    <!-- 错误日志单独输出 -->
    <appender name="ERROR_FILE" class="ch.qos.logback.core.rolling.RollingFileAppender">
        <file>logs/error.log</file>
        <filter class="ch.qos.logback.classic.filter.LevelFilter">
            <level>ERROR</level>
            <onMatch>ACCEPT</onMatch>
            <onMismatch>DENY</onMismatch>
        </filter>
        <rollingPolicy class="ch.qos.logback.core.rolling.TimeBasedRollingPolicy">
            <fileNamePattern>logs/error.%d{yyyy-MM-dd}.log</fileNamePattern>
            <maxHistory>90</maxHistory>
        </rollingPolicy>
        <encoder>
            <pattern>%d{yyyy-MM-dd HH:mm:ss.SSS} [%thread] %-5level %logger{36} - %msg%n</pattern>
        </encoder>
    </appender>

    <!-- 给特定包设置不同级别 -->
    <logger name="com.example.dao" level="DEBUG"/>
    <logger name="org.springframework" level="WARN"/>

    <!-- 根日志：默认 INFO，输出到控制台和文件 -->
    <root level="INFO">
        <appender-ref ref="CONSOLE"/>
        <appender-ref ref="FILE"/>
        <appender-ref ref="ERROR_FILE"/>
    </root>

</configuration>
```

### 配置文件三大核心元素

| 元素           | 作用                          |
| ------------ | --------------------------- |
| **appender** | 日志输出"目的地"（控制台、文件、远程服务器等）    |
| **logger**   | 给特定包/类设置独立的日志级别            |
| **root**     | 默认的日志级别和目的地                 |

### Pattern 占位符速查

| 占位符           | 含义              |
| ------------- | --------------- |
| `%d{...}`     | 时间戳             |
| `%thread`     | 线程名             |
| `%level`      | 日志级别            |
| `%logger{36}` | 类名（最长 36 字符）    |
| `%msg`        | 日志消息            |
| `%n`          | 换行              |
| `%line`       | 行号（性能差，少用）      |

## 八、Spring Boot 中的日志

Spring Boot 默认就用 SLF4J + Logback，配置更简单——**直接在 `application.yml` 里写**：

```yaml
logging:
  level:
    root: INFO
    com.example: DEBUG          # 自己的包开 DEBUG
    org.springframework: WARN   # Spring 框架的日志只看 WARN+

  file:
    name: logs/app.log
    max-size: 100MB
    max-history: 30

  pattern:
    console: "%d{HH:mm:ss.SSS} [%thread] %-5level %logger{36} - %msg%n"
```

> 💡 高级配置仍可放在 `logback-spring.xml`，会被自动加载。

## 九、Lombok @Slf4j —— 一行代码生成 logger

```java
import lombok.extern.slf4j.Slf4j;

@Slf4j   // 自动生成：private static final Logger log = LoggerFactory.getLogger(...)
public class UserService {

    public void login(String username) {
        log.info("用户 {} 登录", username);   // 直接用 log 就行
    }
}
```

引入 Lombok：
```xml
<dependency>
    <groupId>org.projectlombok</groupId>
    <artifactId>lombok</artifactId>
    <version>1.18.32</version>
    <scope>provided</scope>
</dependency>
```

> 💡 实际项目里**几乎所有类都用 `@Slf4j`**，省事又统一。

## 十、MDC —— 给日志加"上下文"

**MDC（Mapped Diagnostic Context）** 让你在每条日志里自动加上下文信息（如请求 ID、用户 ID）。排查问题时**靠它能快速串起一次请求的所有日志**。

```java
import org.slf4j.MDC;

public void handleRequest(String requestId, String userId) {
    MDC.put("requestId", requestId);
    MDC.put("userId", userId);
    try {
        log.info("开始处理请求");   // 日志会自动带上 requestId 和 userId
        // ... 业务逻辑
        log.info("处理完成");
    } finally {
        MDC.clear();   // 必须清除，避免污染下一个请求
    }
}
```

配置 pattern 加上 `%X`：
```xml
<pattern>%d{HH:mm:ss.SSS} [%thread] [%X{requestId}] [%X{userId}] %-5level %logger - %msg%n</pattern>
```

输出：
```
14:23:18.500 [http-nio-8080-exec-1] [req-abc123] [user-456] INFO  UserService - 开始处理请求
14:23:18.520 [http-nio-8080-exec-1] [req-abc123] [user-456] INFO  UserService - 处理完成
```

> 💡 Web 项目通常用**拦截器/过滤器**自动给每个请求设置 MDC。

## 十一、性能小技巧：isXxxEnabled

如果**生成日志参数本身代价大**，可以先判断级别：

```java
// 如果 expensiveOperation() 计算很慢
if (log.isDebugEnabled()) {
    log.debug("详细信息：{}", expensiveOperation());
}
```

> 💡 普通占位符 `log.debug("...{}", x)` 已经是懒拼接，无需 isXxxEnabled。**只有在参数计算特别贵时才需要这个判断。**

## 十二、日志最佳实践

### ✅ 该做的

- **每个类用一个 logger**，名字用类名（`LoggerFactory.getLogger(XxxService.class)`）
- 用 **占位符** `{}`，不用字符串拼接
- 异常**一定**把异常对象传进去（`log.error("...", e)`）
- 重要业务节点打 **INFO**，便于追踪
- 用 **MDC** 串联一次请求的所有日志
- 生产环境用 INFO，开发环境用 DEBUG
- 日志文件**按天/按大小轮转**，定期清理

### ❌ 不该做的

- 不要用 `System.out.println`
- 不要在日志里输出**敏感信息**（密码、身份证、手机号、token）
- 不要在循环里打大量 INFO（容易刷爆磁盘）
- 不要日志级别用错（一切都打 INFO 或一切都打 ERROR）
- 不要捕获异常后只打 message 而丢弃堆栈：

```java
// ❌ 丢失堆栈，定位不到问题
log.error("出错了：" + e.getMessage());

// ✅ 把异常对象传入
log.error("出错了", e);
```

## 十三、敏感信息脱敏示例

```java
// ❌ 危险
log.info("用户登录：username={}, password={}", username, password);

// ✅ 脱敏
log.info("用户登录：username={}", username);
log.info("手机号：{}", maskPhone(phone));   // 138****1234

private String maskPhone(String phone) {
    if (phone == null || phone.length() < 11) return phone;
    return phone.substring(0, 3) + "****" + phone.substring(7);
}
```

## 十四、日志级别在不同环境的建议

| 环境       | 推荐级别        |
| -------- | ----------- |
| 本地开发     | DEBUG       |
| 测试环境     | DEBUG / INFO |
| **生产环境** | **INFO**    |
| 灰度/压测    | INFO（关键路径加 DEBUG）|

> 💡 生产环境如果一时排查问题需要 DEBUG，可以**动态调整日志级别**（Spring Boot 的 `/actuator/loggers` 端点支持），不用重启。

## 十五、各日志框架关系总结

```
你写的代码 → SLF4J 接口  ←——— 这层永远不变
                ↓
       SLF4J 适配桥接器
                ↓
        具体的日志实现（任选其一）
       ┌───────┬────────┬──────┐
    Logback  Log4j2   JUL   ...
```

**业务代码永远只 import `org.slf4j.*`**，永远不要直接 import 具体实现的类。这样将来换底层框架，**业务代码一行不用改**。
