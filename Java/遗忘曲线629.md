# 遗忘曲线算法 · 代码详解

> 涉及文件：
> - `controller/ReviewController.java`（接口层）
> - `service/ReviewService.java`（接口定义层）
> - `service/impl/ReviewServiceImpl.java`（实现层 · 算法核心）

---

## 一、这个功能在干嘛

基于**艾宾浩斯遗忘曲线**做"间隔复习"。核心思想：人对知识的遗忘是先快后慢的，所以复习不该天天来，而要按**越来越长的间隔**安排——刚学完隔 1 天复习，记住了就隔 2 天、4 天、7 天……每复习成功一次就拉长下次间隔，直到彻底掌握；中途忘了就打回最短间隔，从头再来。

整个算法的实现就两个关键点：

1. 一张**间隔天数表** `REVIEW_INTERVALS`，决定"第几阶段、几天后复习"。
2. 一组 **if-else 状态机**（在 `completeReview` 里），决定"答对升级 / 答错重置 / 走完毕业"。

这段代码几乎不含 Stream 和数学运算，主要是 `if-else` 走流程 + MyBatis-Plus 查改数据库，是项目里最好懂的算法之一。

## 二、调用链路（三层是怎么串起来的）

```
前端请求
   │
   ▼
ReviewController          ← 接收 HTTP 请求，解析参数，包装统一响应 Result
   │  调用
   ▼
ReviewService（接口）      ← 只声明"有哪些能力"，不写实现
   │  实现
   ▼
ReviewServiceImpl         ← 真正的算法逻辑：间隔表 + 状态机 + 查改库
   │  访问
   ▼
ReviewPlanMapper / WrongBookMapper / QuestionMapper  ← 操作数据库
```

对外暴露的接口（路由前缀 `/review`）：

| 方法 | URL | 作用 |
|------|-----|------|
| `GET`  | `/review/today/{userId}` | 取今日待复习题目（含逾期） |
| `GET`  | `/review/calendar/{userId}?days=30` | 取未来 N 天复习日历 |
| `POST` | `/review/complete/{planId}?isCorrect=true` | 提交复习结果，自动升阶/重置 |

## 三、Controller 层详解（ReviewController）

控制器只做三件事：**接参数、调服务、包结果**。不写业务逻辑。

```java
@RestController                 // 标记为 REST 控制器，方法返回值自动转 JSON
@RequestMapping("/review")      // 本类所有接口的统一前缀 /review
@Tag(name = "复习计划接口")       // Swagger 文档分组
public class ReviewController {

    @Autowired                  // 由 Spring 自动注入复习服务
    private ReviewService reviewService;
```

- `@RestController`：这个类里每个方法的返回值都会被自动转成 JSON 发给前端。
- `@RequestMapping("/review")`：给本类所有接口加上 `/review` 前缀。
- `@Autowired private ReviewService reviewService;`：把"复习服务"注入进来。注意类型是**接口** `ReviewService`，运行时 Spring 自动塞进实现类 `ReviewServiceImpl` 的对象——这就是"面向接口编程"。

### 1）今日待复习

```java
@GetMapping("/today/{userId}")
public Result<List<ReviewPlan>> getTodayPlans(@PathVariable Integer userId) {
    List<ReviewPlan> plans = reviewService.getTodayReviewPlans(userId);  // 调服务
    return Result.success("今日有 " + plans.size() + " 道题需要复习", plans);
}
```

- `@GetMapping("/today/{userId}")`：映射 `GET /review/today/{userId}`。URL 里的 `{userId}` 是占位。
- `@PathVariable Integer userId`：把 URL 路径里的那段值取出来当参数。访问 `/review/today/5`，`userId` 就是 5。
- `Result.success(...)`：把数据包进统一响应结构（含 code、message、data）再返回。`plans.size()` 是列表长度。

### 2）复习日历

```java
@GetMapping("/calendar/{userId}")
public Result<List<ReviewPlan>> getCalendar(
        @PathVariable Integer userId,
        @RequestParam(defaultValue = "30") int days) {     // 查询参数 ?days=，默认 30
    List<ReviewPlan> plans = reviewService.getReviewCalendar(userId, days);
    return Result.success(plans);
}
```

- `@RequestParam(defaultValue = "30") int days`：接收 URL 问号后的参数 `?days=7`；不传则默认 30。
- 与 `@PathVariable` 的区别：`@PathVariable` 取的是路径里 `/{}` 的部分，`@RequestParam` 取的是 `?key=value` 的部分。

### 3）完成复习（触发状态机）

```java
@PostMapping("/complete/{planId}")
public Result<String> completeReview(
        @PathVariable Integer planId,
        @RequestParam boolean isCorrect) {                 // 本次复习是否答对

    reviewService.completeReview(planId, isCorrect);       // 核心：答对升阶/答错重置

    String message = isCorrect
            ? "回答正确！已安排下一阶段复习"
            : "回答错误，复习阶段已重置，明天再来巩固";
    return Result.success(message, null);
}
```

- `isCorrect ? A : B` 是**三元表达式**："条件成立用 A，否则用 B"，等价于一个简短的 if-else。
- 真正的升阶/重置逻辑全在 `reviewService.completeReview(...)` 里，控制器只负责调它、再返回一句提示。

## 四、Service 接口层详解（ReviewService）

接口层**只声明能力，不写实现**。作用是把"做什么"和"怎么做"分开，方便替换实现、方便测试。

```java
public interface ReviewService {

    /** 答题后触发复习计划（答错加入错题本 + 创建复习计划） */
    void processAnswer(UserAnswer answer);

    /** 完成复习后更新计划（答对升阶 / 答错重置） */
    void completeReview(Integer planId, boolean isCorrect);

    /** 获取用户今日待复习的题目列表 */
    List<ReviewPlan> getTodayReviewPlans(Integer userId);

    /** 获取用户未来 N 天的复习日历数据 */
    List<ReviewPlan> getReviewCalendar(Integer userId, int days);
}
```

这里只有方法签名（名字、参数、返回类型），没有方法体。`ReviewServiceImpl` 必须把这四个方法全部实现。Controller 调的是接口，运行时实际跑的是实现类。

## 五、Impl 层详解（ReviewServiceImpl · 算法核心）

### 1）两个常量：整个算法的地基

```java
private static final int[] REVIEW_INTERVALS = {0, 1, 2, 4, 7, 15, 30};
private static final int MAX_STAGE = 6;
```

`REVIEW_INTERVALS` 是**间隔天数表**，用"阶段号"当下标去取：

| 阶段 | 取值写法 | 间隔天数 |
|------|----------|----------|
| 1 | `REVIEW_INTERVALS[1]` | 1 天 |
| 2 | `REVIEW_INTERVALS[2]` | 2 天 |
| 3 | `REVIEW_INTERVALS[3]` | 4 天 |
| 4 | `REVIEW_INTERVALS[4]` | 7 天 |
| 5 | `REVIEW_INTERVALS[5]` | 15 天 |
| 6 | `REVIEW_INTERVALS[6]` | 30 天 |

- 下标 0 那个 `0` 是**占位用、不使用**的废值。因为阶段从 1 开始数，留个第 0 格占位，能让"阶段号 = 下标"，取间隔时直接 `REVIEW_INTERVALS[stage]`，不用 `stage-1`，省脑子。
- `MAX_STAGE = 6`：最高阶段。走完第 6 阶段还答对，就算彻底掌握。
- `static final` = 写死的常量、全局共用一份，理解为"配置"即可。

### 2）触发点一：答完题 `processAnswer`

```java
@Override
@Transactional                                    // 多次写库要么全成功要么全回滚
public void processAnswer(UserAnswer answer) {
    Integer userId = answer.getUserId();
    Integer questionId = answer.getQuestionId();
    boolean isCorrect = answer.getIsCorrect() == 1;   // 数据库存 1/0，转成 true/false

    if (!isCorrect) {                              // 答错了
        addToWrongBook(userId, questionId);            // 进错题本
        createReviewPlan(userId, questionId, 1);       // 从第 1 阶段建复习计划
    } else {                                       // 答对了
        markIfMastered(userId, questionId);            // 检查掌握（目前是空壳，预留扩展）
    }
}
```

- `answer.getIsCorrect() == 1`：数据库里对错存的是数字 1/0，`== 1` 把它转成 `true`/`false` 方便判断。
- 逻辑：**答错 → 进错题本 + 建一条第 1 阶段计划；答对 → 走个检查（该方法目前没干实事）。**
- `createReviewPlan(..., 1)` 最后那个 `1` 就是"从第 1 阶段开始"。
- `@Transactional`：保证这个方法里的多次写库操作（加错题本 + 建计划）作为一个整体，要么全成功，要么一起回滚。

### 3）核心方法：建复习计划 `createReviewPlan`

间隔表就在这里被用上：

```java
private void createReviewPlan(Integer userId, Integer questionId, int stage) {
    // 先把这道题旧的、没做完的计划作废，避免同一题堆好几条
    reviewPlanMapper.update(null,
            new LambdaUpdateWrapper<ReviewPlan>()
                    .eq(ReviewPlan::getUserId, userId)         // 条件：这个用户
                    .eq(ReviewPlan::getQuestionId, questionId) // 条件：这道题
                    .eq(ReviewPlan::getIsCompleted, 0)         // 条件：还没完成的
                    .set(ReviewPlan::getIsCompleted, 1));      // 全部标记成已完成（=作废）

    // ★关键★ 复习日期 = 今天 + 该阶段的间隔天数
    LocalDate reviewDate = LocalDate.now().plusDays(REVIEW_INTERVALS[stage]);

    Question question = questionMapper.selectById(questionId); // 查题目（为了拿它的知识点）

    ReviewPlan plan = new ReviewPlan();
    plan.setUserId(userId);
    plan.setQuestionId(questionId);
    plan.setKnowledgeId(question != null ? question.getKnowledgeId() : 0);  // 题在则取知识点，不在给 0
    plan.setReviewDate(reviewDate);
    plan.setReviewStage(stage);
    plan.setIsCompleted(0);                        // 状态：未完成
    reviewPlanMapper.insert(plan);                 // 存进数据库
}
```

两个重点：

**第一段是"先作废旧的"。** `LambdaUpdateWrapper` 是 MyBatis-Plus 的**更新条件构造器**（和查询用的 `LambdaQueryWrapper` 是兄弟）。这里把"这个用户、这道题、还没完成的"计划，全部 `.set(...IsCompleted, 1)` 标成已完成。目的：防止同一道题同时存在好几条活跃计划，把复习列表刷爆。

**最关键的一行：**

```java
LocalDate reviewDate = LocalDate.now().plusDays(REVIEW_INTERVALS[stage]);
```

念成大白话："复习日期 = 今天（`LocalDate.now()`）往后加（`plusDays`）`REVIEW_INTERVALS[stage]` 天。" `stage=1` 就是今天 +1 天；`stage=4` 就是今天 +7 天。**间隔表在这里被 `[stage]` 取出来用上了。**

`question != null ? ... : 0` 是三元表达式：题目还在就拿它的知识点 ID，被删了就填 0，防止空指针。

### 4）触发点二：完成复习 `completeReview`（状态机的灵魂）

这是体现"答对升级、答错重来、走完毕业"的地方：

```java
@Override
@Transactional
public void completeReview(Integer planId, boolean isCorrect) {
    ReviewPlan plan = reviewPlanMapper.selectById(planId);   // 取出这条计划
    if (plan == null || plan.getIsCompleted() == 1) {        // 不存在，或已经做过了
        return;                                              // 直接返回，避免重复处理
    }

    // 先把当前这条计划标记成"做完了"
    plan.setIsCompleted(1);
    plan.setCompletedTime(LocalDateTime.now());
    plan.setIsCorrect(isCorrect ? 1 : 0);                    // 记录本次复习对错
    reviewPlanMapper.updateById(plan);

    if (isCorrect) {                                         // ===== 复习答对 =====
        int nextStage = plan.getReviewStage() + 1;              // 阶段 +1（升级）
        if (nextStage > MAX_STAGE) {                            // 升过头（已超第 6 阶段）
            markAsMastered(plan.getUserId(), plan.getQuestionId());  // 标记已掌握，毕业
        } else {                                                // 还没到顶
            createReviewPlan(plan.getUserId(), plan.getQuestionId(), nextStage);  // 建下一阶段计划（间隔更长）
        }
    } else {                                                // ===== 复习答错 =====
        addToWrongBook(plan.getUserId(), plan.getQuestionId());      // 重新进错题本
        createReviewPlan(plan.getUserId(), plan.getQuestionId(), 1); // 阶段重置为 1（从头再来）
    }
}
```

- 开头 `if (plan == null || plan.getIsCompleted() == 1) return;` 是**防御性检查**：计划不存在、或已经做过，就别处理——避免重复升级或报错。`||` 是"或者"，两个条件满足任一个就返回。
- **答对**：`nextStage = 当前阶段 + 1`（升一级）。若 `nextStage > 6`（第 6 阶段还答对），调 `markAsMastered` 毕业；否则 `createReviewPlan(..., nextStage)` 建下一阶段——传进去的是 `nextStage`，于是间隔表取出**更长的间隔**。
- **答错**：`createReviewPlan(..., 1)`，最后那个 `1` 就是**打回第 1 阶段**，间隔归零，从头再来。
- `isCorrect ? 1 : 0`：把 true/false 转回数字 1/0 存库（和前面 `== 1` 方向相反）。

### 5）取今日待复习 `getTodayReviewPlans`

```java
List<ReviewPlan> plans = reviewPlanMapper.selectList(
        new LambdaQueryWrapper<ReviewPlan>()
                .eq(ReviewPlan::getUserId, userId)          // 这个用户
                .eq(ReviewPlan::getIsCompleted, 0)          // 还没做的
                .le(ReviewPlan::getReviewDate, LocalDate.now())  // 复习日期 <= 今天
                .orderByAsc(ReviewPlan::getReviewDate));    // 按日期排
```

注意 `.le(...)` 是 "less than or equal（小于等于）"，条件是**复习日期 ≤ 今天**，而不是 `= 今天`。意味着**昨天该做但没做的，今天还会一直显示**，不会因为过了那天就漏掉——这是个贴心设计。

后面有个 for 循环给每条计划补上题目内容（查一次题目塞进去），方便前端直接显示题干，逻辑简单不展开。

### 6）两个辅助方法（简述）

- `addToWrongBook(userId, questionId)`：加入错题本。已有该错题则错误次数 +1、并把"已掌握"重置为未掌握；没有则新建一条，错误次数初始为 1。
- `markAsMastered(userId, questionId)`：把错题本里这道题标记为已掌握，并记录掌握时间。
- `markIfMastered(...)`：答对时调用的检查方法，目前是**空壳**（预留扩展点）。也就是说"答对即掌握"暂未实现，掌握判定完全靠走完 6 阶段复习流程。

## 六、整段流程串起来

```
答错一道题
  → 进错题本 + 建【第1阶段】计划（明天复习）

明天来复习：
  ├ 答对 → 升【第2阶段】(2天后) → 对 → 【第3阶段】(4天后) → … → 第6阶段答对 → 毕业（已掌握）
  └ 答错 → 打回【第1阶段】（又从明天开始）
```

核心机制就两点：

1. **间隔表 `REVIEW_INTERVALS[stage]` 决定"几天后复习"**。
2. **`completeReview` 里的 if-else 决定"阶段是升、是毕业、还是打回 1"**。

其余全是查库、存库的常规操作。

## 七、关键设计点与可优化项

**设计得好的地方：**

- 建新计划前先作废旧的未完成计划，避免同题堆积多条活跃计划。
- 今日待复习用 `≤ 今天` 而非 `= 今天`，逾期任务不会丢。
- 写库方法加了 `@Transactional`，保证"加错题本 + 建计划"的原子性。
- 间隔表用下标 0 占位，让阶段号直接当下标，代码更干净。

**可优化项：**

- `markIfMastered` 是空壳，"答对即掌握"未实现，目前只能靠走完全程毕业。可按需补上"连续答对 N 次即掌握"的逻辑。
- 间隔表和阈值都写死在代码里，若要做成可配置（不同用户不同节奏），可抽到配置或数据库。
